# Agent辅助证明Delaunay三角剖分Stretch Factor上界：系统架构设计

## 1 问题背景

### 1.1 数学问题

Delaunay三角剖分的stretch factor上界是计算几何的长期开放问题。当前最优上界为Xia (2013)证明的 ρ = 1.998，已知最优下界为1.5932 (Xia & Zhang, 2011)。

Xia的证明引入势函数 Φ_O 和目标函数 Υ_O(u,v) = |P_O(u,v)| − λ|D_O(u,v)| + Φ_O，通过证明两个核心引理得到上界：

- **Lemma 2**：Υ_O(u,v) < 0 对所有链 O 和终点 u,v 成立
- **Lemma 3**：对极端链 O*，Φ_{O*} ≥ −C · |P_{O*}(u,v)|

最终上界 ρ = λ/(1−C)。改进上界的核心在于设计更好的势函数 Φ'_O 使常数 C 更小。

### 1.2 势函数必须满足的六个约束

| 编号 | 名称 | 内容 | 用于 |
|------|------|------|------|
| N1 | 基础可行性 | Φ\|_{n=1} ≤ (λ−π/2)·\|D\|，理想为0 | Lemma 2 基础情形 |
| N2 | 末端单调性 | Φ_O ≤ Φ_{O_{1,n-1}} | Prop 2，**不可绕过** |
| N3 | 分裂次可加性 | Φ_O ≤ Φ_{O_{1,j+1}} + Φ_{O_{j+1,n}} | Prop 3，**不可绕过** |
| D4 | 终点无关或凸性 | Φ 不依赖 v，或关于 v 位置凸 | Prop 4, 6 |
| Q5 | 偏导数足够负 | ∂Φ/∂X_{o_n} 使 g_i(α) < 0 | Prop 7 |
| Q6 | 极端链下界 | Φ_{O*} ≥ −C'·\|P\|，C' 越小越好 | Lemma 3 |

**核心张力**：Q5要求Φ变化快（偏导数够负），Q6要求Φ不能太负。两者方向相反，最优 ρ 对应两者同时紧绷的临界点。

### 1.3 原证明中的代码验证角色

通读全文后，**唯一依赖代码验证的步骤是Proposition 7 (Section 7.5)**：验证四个单变量函数 g₁(α), g₂(α), g₃(α), g₄(α) < 0。作者使用Lipschitz常数(≤16)配合Piyavskii递归二分算法完成。其余全部步骤为纯数学推导。

对于新势函数，需要代码验证的大概率也集中在**新 g' 函数是否 < 0**这一步。

### 1.4 关于Lean形式化的立场

完整Lean形式化当前不可行且非必要：证明瓶颈在势函数的创造性设计，非逻辑严密性；几何对象的形式化需先建库，工作量巨大；Piyavskii数值验证在Lean中实现极为痛苦。

**务实路线**：Python/Mathematica做数值验证，SymPy做符号辅助，仅对最终关键新引理考虑Lean验证。

## 2 系统架构

### 2.1 三层结构概览

系统分为三层：**元控制层**负责全局调度与记忆管理；**自主探索层**是一个自治流水线，完成从势函数设计到证明验证的完整闭环；**验证工具层**提供数值和符号计算能力。

关键设计原则：**探索层内部自主运行**。势函数设计师主动调用证明规划师，规划师调用证明撰写者，撰写者与正确性检查员形成审查回路。仅当某条路径被剪枝或通过全部审查时，才将结果返回元控制层。Orchestrator不介入探索的中间过程。

```
元控制层
┌─────────────────────────────────────────────┐
│  Orchestrator（主脑）  ←→  Memory Manager   │
└───────────┬──────────────────────────▲──────┘
            │ 方向指令                 │ 剪枝/通过
            ▼                         │
自主探索层（自治流水线）                │
┌─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┤
│                                     │
│  势函数设计师                        │
│       │ 调用                        │
│       ▼                             │
│  证明策略规划师                      │
│       │ 调用                        │
│       ▼                             │
│  证明撰写者  ──提交──→  正确性检查员  │
│       ▲                    │        │
│       └──── 修改(≤N次) ────┘        │
│                                     │
└─ ─ ─ ─ ─ ─ ─│─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┘
               │ 按需调用
               ▼
验证工具层
┌─────────────────────────────────────────────┐
│  数值验证器  │  符号计算器  │  领域文献RAG   │
└─────────────────────────────────────────────┘
```

### 2.2 元控制层

#### 2.2.1 Orchestrator（主脑）

**职责**：全局决策者，不参与探索细节。

**输入**：来自自主探索层的终态报告（剪枝报告或全通过报告）。

**决策内容**：
- 收到**通过报告**：记录成功候选，决定是继续优化还是转入证明完善阶段
- 收到**剪枝报告**：分析失败原因，决定下一步方向
  - 如果失败因某个性质的根本障碍：建议设计师换势函数族
  - 如果失败因参数范围：建议设计师在相邻参数空间重试
  - 如果多个候选败在同一性质：触发对该性质的深入分析

**输出**：给势函数设计师的方向指令，如：
- "继续探索形如 Φ = f(r_n,r_1) − Σw(H_i,V_i) 的候选，但尝试非线性权重"
- "Section 6中uv∩O_n的思路值得尝试，优先验证其N2可行性"
- "暂停新候选探索，对当前最优的 Φ'_7 进入证明完善阶段"

#### 2.2.2 Memory Manager（记忆管理员）

**职责**：维护结构化探索历史，供所有Agent查询。

**核心数据结构——探索树**：

```
{
  candidate_id: "Φ'_3",
  form: "φ(r_n − r_1) − (φ/3)Σ(2.5·H_i + 0.8·V_i)",
  derived_from: "Φ'_1 (调整了H:V权重比)",
  property_status: {
    N1: { status: "pass", note: "Φ|_{n=1} = 0" },
    N2: { status: "fail", note: "2.5H_n + 0.8V_n < 3(r_n−r_{n-1})在r_n≈r_{n-1}时不成立" },
    N3: { status: "untested" },
    ...
  },
  pruned: true,
  pruned_reason: "N2不可修复：权重比改变破坏了关键不等式(5)的推导",
  exploration_log: [ ... ]  // 完整的设计→规划→撰写→检查过程记录
}
```

**同时维护**：
- 领域知识库（与RAG共享）
- 性质间依赖关系的结构化知识（如"N2失败则Prop 2不可用"）
- 累积的设计启发式（如"添加 v 依赖项大概率破坏D4"）

### 2.3 自主探索层（自治流水线）

这是系统的核心工作层。四个角色形成一条自治流水线：设计师发起、规划师规划、撰写者执行、检查员把关。

#### 2.3.1 势函数设计师（Potential Function Designer）

**职责**：提出新的势函数候选 Φ'。

**灵感来源**（按优先级）：
1. Orchestrator的方向指令
2. LLM的数学直觉（基于对势函数理论、几何不等式、变分法等的理解）
3. 领域文献RAG检索（查询类似问题中使用的摊还分析技巧）
4. Memory Manager中的历史经验（特别是近期失败的教训）

**输出**：一个精确的数学表达式 + 初步直觉分析（哪些性质可能满足/困难）。

**后续动作**：设计师直接调用证明策略规划师，将候选交付评估。无需等待Orchestrator批准。

#### 2.3.2 证明策略规划师（Proof Strategy Planner）

**职责**：对给定候选 Φ'，制定完整的证明路线图。

**输入**：来自设计师的候选 Φ' 及其初步分析。

**输出**：结构化证明计划，包含：

```
proof_plan(Φ'_k):
  reusable:    [Prop 2, Prop 3, Prop 8, Prop 9]  // 可复用原证明
  needs_redo:  [Lemma 1, Prop 4, Prop 6, Prop 7]  // 需要重写
  unchanged:   [Prop 5]                            // 不受影响
  priority:    [N2, N3, D4, Q5, Q6]                // 建议验证顺序
  risk_notes:  "主要风险在D4——新Φ依赖v，需证凸性"
  estimated_C: "≈ 0.12 → ρ ≈ 1.97"               // 粗估
```

**建议验证顺序的逻辑**：先验证必要性质（N2, N3），再验证高度期望性质（D4），最后处理定量约束（Q5, Q6）。如果N2/N3就失败了，后续工作全部省略。

**后续动作**：规划师直接调用证明撰写者，按优先级逐个交付Proposition。

#### 2.3.3 证明撰写者（Proof Writer）

**职责**：对规划师指定的每个Proposition，撰写详细的数学证明。

**工作模式**：
- 接收任务（如"证明 Φ'_k 满足N2"）
- 撰写推导过程
- **按需调用验证工具**：
  - 数值验证器：验证新 g' 函数是否 < 0
  - 符号计算器：计算 ∂Φ'/∂X_{o_n}、化简表达式
- 将完成的证明提交给正确性检查员

#### 2.3.4 正确性检查员（Correctness Checker / 剪枝器）

**职责**：以Proposition为粒度审查证明正确性。

**审查要点**：
- 每步推导的逻辑合法性
- 引用的引理/命题是否正确适用
- 不等式方向和边界条件
- 隐含假设是否成立

**错误处理机制**：

```
检查员发现错误
    │
    ├─→ 告诉撰写者具体错误位置和原因
    │     │
    │     ├── 撰写者修改后重新提交（计入重试计数）
    │     │
    │     └── 重试次数达到上限 N（建议N=3）
    │           │
    │           └── 该Proposition被标记为失败
    │                 │
    │                 └── 整个候选被剪枝
    │                       │
    │                       └── 打包剪枝报告，返回Orchestrator
    │
    └─→ 无错误：该Proposition通过
          │
          └── 继续下一个Proposition（由规划师安排）
               │
               └── 全部通过 → 打包通过报告，返回Orchestrator
```

### 2.4 验证工具层

#### 2.4.1 数值验证器

**实现**：Python脚本，基于Lipschitz优化的Piyavskii算法。

**功能**：验证形如 g'(α) < 0 的不等式在指定区间上严格成立。

**接口**：
```python
verify(g_func, domain, lipschitz_bound) → (pass, max_value) | (fail, counterexample_α)
```

**被谁调用**：证明撰写者（在处理Prop 7对应部分时）。

#### 2.4.2 符号计算器

**实现**：SymPy。

**功能**：偏导数计算、三角恒等式化简、表达式展开与合并。

**典型任务**：
- 计算 ∂Φ'/∂X_{o_n}（对应原证明Section 7.3的推导）
- 化简 f(α,β,γ) 表达式以确定Lipschitz常数
- 验证代数恒等式

**被谁调用**：证明撰写者。

#### 2.4.3 领域文献RAG

**实现**：小型向量数据库，包含核心论文的分块嵌入。

**内容**：
- Xia 2013 原论文（核心参考）
- Xia & Zhang 2011（下界构造，含极端链序列的性质）
- Tan et al. 2024（凸情形改进至1.77，可能有可借鉴的势函数技巧）
- Bose et al. 系列（spanner的一般性工具）
- 相关的几何不等式工具集

**被谁调用**：势函数设计师（获取设计灵感）。

## 3 工作流程

### 3.1 正常流程（候选通过）

```
Orchestrator → 势函数设计师:
  "探索在Φ中引入r_i的二次项的可能性"

  势函数设计师（查询RAG获取灵感，提出候选）
    → 证明策略规划师:
        "候选 Φ'_8 = φ(r_n−r_1) − (φ/3)Σ(2H_i+V_i) + c·Σ(r_i−r_{i-1})²
         初步判断：N1=0因为n=1时和为空，N2需验证，N3因二次项可能不可加"

      证明策略规划师:
        "路线图：先验N3（最可能失败），再验N2，然后D4/Q5/Q6。
         N3验证：需要 c·(r_n−r_1)² ≤ c·(r_{j+1}−r_1)² + c·(r_n−r_{j+1})²
         这等价于 2(r_{j+1}−r_1)(r_n−r_{j+1}) ≤ 0，不成立。N3失败。"

        → 直接剪枝（无需调用撰写者，规划阶段即可判定）

  剪枝报告 → Orchestrator:
    "Φ'_8 被剪枝。原因：二次项 Σ(r_i−r_{i-1})² 破坏N3次可加性。
     教训：势函数中的凸函数项（如平方项）不满足次可加性。"

Orchestrator → Memory Manager: 存储
Orchestrator → 势函数设计师:
  "避免凸函数形式的附加项。考虑凹函数或线性项。"
```

### 3.2 正常流程（需完整证明的候选）

```
势函数设计师 → 证明策略规划师:
  "候选 Φ'_12 = φ(r_n−r_1) − (φ'/3)Σ(1.8·H_i + 1.2·V_i)，其中φ'略不同于φ"

证明策略规划师:
  "路线图：N2需重证（系数改变影响不等式(5)），其余结构不变。
   优先级：N2 → N3 → Q5 → Q6"

  → 证明撰写者: "证明N2"

    证明撰写者:
      "Φ_O − Φ_{O_{1,n-1}} = φ'(r_n−r_{n-1}) − (φ'/3)(1.8·H_n + 1.2·V_n)
       需要 1.8·H_n + 1.2·V_n ≥ 3(r_n−r_{n-1})
       由 H_n = ||o_n o_{n-1}|| ≥ r_n−r_{n-1} 和 V_n ≥ |r_n−r_{n-1}|
       得 1.8(r_n−r_{n-1}) + 1.2(r_n−r_{n-1}) = 3(r_n−r_{n-1}) ✓"

    → 正确性检查员审查: "通过。"

  → 证明撰写者: "证明Q5（计算新偏导数并验证g'函数）"

    证明撰写者 → 符号计算器:
      "计算 ∂Φ'_12/∂X_{o_n}"

    证明撰写者 → 数值验证器:
      "验证 g'_1(α),...,g'_4(α) < 0"

    数值验证器: "g'_3(α) 在 α≈0.47 处取值 +0.002 > 0，验证失败"

    → 正确性检查员: "Q5 不满足（数值反例）"

  剪枝报告 → Orchestrator:
    "Φ'_12 被剪枝。N2/N3通过，但Q5失败：g'_3 在α∈(0.4,0.5)附近为正。
     意味着偏导数项在小角度区域不够负。"

Orchestrator → 势函数设计师:
  "Φ'_12的H:V=1.8:1.2方向可行（N2通过），但Q5在小α处失败。
   尝试在α<π/2区域增加∂Φ/∂X的负值——可能需要在小角度时给H更大的权重。"
```

### 3.3 检查员触发修改回路

```
证明撰写者提交Prop 4证明 → 正确性检查员:
  "第3步有误：声称 ℓ(uv,O_n) 关于 |A_n| 是凸函数，
   但未考虑 v 移动时 uv 方向也在变化，导致 ℓ 的二阶导可能为负。"

  → 撰写者修改（尝试1）:
    "改用分段论证：当 β−γ > 0 时凸，当 β−γ < 0 时..."

  → 检查员: "分段论证的边界处连续性未验证。"

  → 撰写者修改（尝试2）:
    "直接计算 d²ℓ/d|A_n|² 的精确表达式..."
    → 调用符号计算器
    "得到 d²ℓ/d|A_n|² = (sin²α − β²cos²α) / r_n·(...) ≥ 0 当 β ≤ sinα。✓"

  → 检查员: "通过。" ✓
```

## 4 设计决策总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 探索层是否自治 | **是，仅终态上报** | 减少Orchestrator瓶颈，加速探索循环 |
| 验证策略 | **Python数值验证** | 唯一需要代码的就是g函数，Lean性价比极低 |
| 检查粒度 | **Proposition级别** | 平衡效率与可靠性 |
| 错误回流目标 | **先回撰写者，N次失败再剪枝** | 多数错误是表述不精确而非方向错误 |
| 设计师灵感来源 | **LLM直觉 + 小型领域RAG** | 势函数设计是创造性工作 |
| 谁调用验证工具 | **证明撰写者按需调用** | 工具是撰写过程的一部分，非独立环节 |
| 规划师能否直接剪枝 | **可以（如N3显然失败）** | 避免不必要的撰写工作 |

## 5 约束速查表

供系统中所有Agent共享的参考：

| 编号 | 性质 | 使用位置 | 必要性 | 当前设计如何满足 |
|------|------|----------|--------|-----------------|
| N1 | Φ\|_{n=1} ≤ (λ−π/2)·\|D\| | Lemma 2基础 | 必要 | Φ\|_{n=1}=0 |
| N2 | 去末端盘不增 | Prop 2 | **不可绕过** | 2H_n+V_n ≥ 3(r_n−r_{n-1}) |
| N3 | 内部分裂次可加 | Prop 3 | **不可绕过** | Φ精确可加 |
| D4 | 不依赖v或关于v凸 | Prop 4, 6 | 高度期望 | 完全不依赖v |
| Q5 | g_i(α) < 0 | Prop 7 | 定量必须 | 数值验证通过 |
| Q6 | Φ_{O*} ≥ −C·\|P\| | Lemma 3 | 定量必须 | C=√5φ/3 |

**优化目标**：在N1-N3、D4（或弱化形式）、Q5全部满足的前提下，最小化 C，得到 ρ = λ/(1−C)。
